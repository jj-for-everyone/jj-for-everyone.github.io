# Undoing mistakes

```admonish tip title="Welcome to level 2 !" 
If you took a break after finishing the previous level, remember that you can [restore the state](./introduction.md#restoring-your-progress) of your example repo in case you lost it.
```

What we've learned so far can go a long way - until something goes wrong.
For example, assume you ran `jj commit` and hastily entered a sloppy commit message.
Now you'd like to go back and improve it.

With our current skill set, it's impossible to edit the description of a commit after we ran `jj commit`.
But what if we could simply **undo** a change we made to the repository?
Travel back in time?
Turns out we can, because Jujutsu stores not only a version history of our project in the form of commits, but it also keeps a log of all operations that were performed on the repository itself.
This is aptly named the **operation log**.
Several commands related to this log are available via `jj op`.

## Inspecting the operation log

First of all, let's get a feel for this operation log by taking a look at it.
We can display it by running the command `jj op log`.
Since we've accumulated a lot of operations already, you may want to limit the number of entries shown with `--limit <number>` or `-n` for short.

Interesting!
I haven't talked much about how the changes we make to files automatically end up in a commit.
Jujutsu doesn't run in the background, so it doesn't actually know about new changes you make until you run it the next time.
A "snapshot" of all your files is taken whenever you run Jujutsu.
The specific command doesn't matter.
As we can see, even the command `jj op log` can cause a snapshot to be taken.

Now we want to describe the changes in the parent commit of the working copy.
We will accomplish this by **undoing** the operation that created the new commit.
There are better ways to achieve the same task (which we'll learn about later), but sometimes it's easiest to just take a step back and start over.
The command we need is aptly named:

```sh

```

## Restoring the repository to any previous state

The `undo` command works great if you just want to undo the most recent operation.
What if you want to undo an operation that is slightly older?
Can you just run `undo` multiple times, similar how the "undo" feature of graphical applications typically works?

```sh

jj undo
```

Clearly that didn't work.
An undo operation that's immediately followed by another undo operation is useless, they cancel each other out.

What if we made multiple mistakes that we need to undo?
In that case, use `jj op restore <ID>` to reset the state of the repository to any past state.
Let's try a really old one:

```sh
jj op restore bbe51630d935 # <- pick any old operation ID
```
